You are populating a sublocation with characters who belong there. There are two sources of characters: existing rows you can REUSE here, or fresh strangers you can SPAWN.

Input:
- LOCATION: where you're populating. Name, description, and any kind/subrole hint.
- PARENT: the containing location (usually a city). Name and description, for flavor context. May be null for top-level locations.
- ALREADY_PRESENT: characters already at this location. You may NOT duplicate their names.
- CANDIDATES: existing characters living at this sublocation's parent city or its other sublocations. For each: their character_id, name, current subrole, `dormant` flag, and most recent events. You may REUSE them — relocating them into this scene with an updated subrole — if they plausibly fit the new role. `dormant: true` candidates are named historicals from past events (genesis backstory) who haven't surfaced in play yet; reusing one wakes them. `dormant: false` candidates are already-active local NPCs.
- TARGET_COUNT: total characters we'd like present at this location after materialization (already counts ALREADY_PRESENT).
- SLOTS_TO_FILL: TARGET_COUNT - ALREADY_PRESENT.count. The upper bound for new characters (reuse + spawn combined).

Output MUST be a JSON object of this exact shape:
{
  "reuse": [
    {
      "character_id": 17,
      "subrole": "barkeep",
      "properties": {
        "personality": "stoic",
        "disposition": "wary of strangers"
      }
    }
  ],
  "spawn": [
    {
      "subrole": "drunk_patron",
      "properties": {
        "personality": "morose"
      }
    }
  ]
}

HARD RULES:
- "reuse" and "spawn" are arrays. Both MAY be empty.
- Every "character_id" in "reuse" MUST appear in CANDIDATES. No inventing reuses.
- Do NOT supply a "name" field for spawn entries. The engine assigns culturally-coherent names mechanically from the kingdom's name pool. Any name you write is ignored. Describe the character's role via "subrole" + "properties" only. This is the load-bearing rule that keeps the world out of generic-fantasy-name purgatory (no more Elara, no more Silas).
- "subrole" is a non-empty free-text archetype string ("barkeep", "guard", "drunk_patron", "smith"). For reuse, this OVERWRITES the candidate's current subrole — only set if the new role differs meaningfully.
- "properties" is an object. May contain free-text fields like "personality", "disposition", "mood". May be an empty object. For reuse, properties MERGE into the existing character's properties; for spawn, they're the initial values. The system also clears `dormant` on reuse automatically — you do not need to manage it.
- Combined count of reuse + spawn MUST NOT exceed SLOTS_TO_FILL.
- Preference order: REUSE-ACTIVE > REUSE-DORMANT > SPAWN. An already-active local NPC with rich history at this city is the best fit; a dormant historical who plausibly could still be here is the next-best; spawning fresh is the last resort. The point: every named historical from genesis has a real row, so the player can find them by walking into the right place — that only works if you actually pick them when they fit.
- Consider each candidate's history when deciding. If their events suggest they died, were exiled, or moved on, do NOT reuse them. Leave them as-is. Dormant historicals especially — only wake one if their events are recent enough that they could plausibly still be around (the founder is still alive, the missing courier never returned, the journeyman who left for the capital is back in town).
- Characters should fit LOCATION's description and kind. A tavern gets a barkeep and patrons, not a guild treasurer. Guildhouses get guild members. Warehouses get laborers or guards.

Not every slot must be filled. A quiet, remote, or depopulated sublocation may legitimately have only one or two characters even if SLOTS_TO_FILL is higher. Choose what fits the scene. Return ONLY the JSON object. No markdown, no explanation.
